          37→## Summary Entries
          38→
          39→```json
          40→{
          41→  "type": "summary",
          42→  "summary": "Human-readable session description",
          43→  "leafUuid": "uuid-of-message-when-summary-was-created"

          46→
          47→| Field | Description |
          48→|-------|-------------|
          49→| `summary` | Text description of the session |

          51→
          52→### Summary Behavior
          53→
          54→**When written**: Summaries are generated when a session exits (observable by preceding "Goodbye!" local command output).
          55→
          56→**Location**: Usually at the start of the file, but can appear in clusters throughout the file (e.g., after session resume/exit cycles).
          57→
          58→**Multiple summaries**: A file may contain multiple summary entries with different `leafUuid` values, representing the evolving session description over time.
          59→
          60→### Session Resume Behavior (Intended Design)
          61→
          62→When using `/resume` to continue a previous session:
          63→
          64→1. **New "pointer" file created**: Contains only a summary entry + file-history-snapshots, with NO actual conversation messages
          65→2. **Original file extended**: The resumed conversation is appended to the original session file
          66→3. **`leafUuid` cross-reference**: The summary in the new file points to the exit message in the original file (the point where the session was exited before resume)
          67→
          68→This is an **intended** cross-file reference - the new file serves as a bookmark/pointer to where the conversation continues in the original file.
          69→
          70→### Cross-Session Summary Contamination (Known Bug)
          71→
          72→As of January 2025, `/resume` appears to be the only intended behavior that creates cross-file `leafUuid` references. Other cross-file references are likely bug contamination.
          73→
          74→There is a known bug ([#2597](https://github.com/anthropics/claude-code/issues/2597), as of v2.0.76) where summaries appear in the wrong session file:
          75→
          76→1. On startup, Claude Code scans all session files in the project directory
          77→2. Generates summaries for conversation chains that lack them
          78→3. **Bug**: Writes ALL generated summaries to the current session file instead of their original files
          79→
          80→**Prevalence**: In one analyzed dataset (v2.0.76), ~80% of summary entries were affected (184/228 summaries with cross-file `leafUuid` references that weren't from `/resume`).
          81→
          82→**Symptoms**:
          83→- `leafUuid` pointing to message UUIDs in unrelated session files
          84→- "Stub" files containing only summaries and file-history-snapshots, with no actual messages
          85→- Multiple unrelated sessions sharing the same inherited summaries
          86→
          87→### Distinguishing Intended vs Bug Behavior
          88→
          89→| Behavior | Source | Characteristics |
          90→|----------|--------|-----------------|
          91→| **Intended** (resume) | `/resume` command | New file has 1 summary pointing to exit message of related session; original file contains the continued conversation |
          92→| **Bug** (contamination) | Startup summary generation | Multiple unrelated summaries; `leafUuid` points to messages in completely unrelated sessions |
          93→
          94→### Finding the Correct Summary for a Session
          95→
          96→To identify legitimate summaries:
          97→
          98→1. **For regular sessions**: Find summaries where `leafUuid` exists as a `uuid` within the same file
          99→2. **For resume pointer files**: The file will have only 1 summary + file-history-snapshots, no messages; the `leafUuid` points to the original session being resumed
         100→3. **Contaminated summaries**: Multiple summaries pointing to various unrelated sessions are likely bug contamination

         594→## Conversation Branching
         595→
         596→Branching occurs when users rewind (`/rewind`) or retry messages.
         597→
         598→### Structure
         599→
         600→- Multiple messages can share the same `parentUuid`
         601→- Each branch continues with its own chain of `uuid` → `parentUuid` links
         602→- All branches are stored chronologically in the file
         603→- No explicit "active branch" marker
         604→
         605→### Identifying Branches
         606→
         607→```
         608→Message A (uuid: aaa)
         609→    ├── Message B1 (parentUuid: aaa, uuid: bbb1)  ← Branch 1
         610→    │       └── Message C1 (parentUuid: bbb1)
         611→    └── Message B2 (parentUuid: aaa, uuid: bbb2)  ← Branch 2 (rewind)
         612→            └── Message C2 (parentUuid: bbb2)
         613→```
         614→
         615→### Finding the Active Branch
         616→
         617→1. Build a tree from `parentUuid` relationships
         618→2. Find all "leaves" (messages with no children)
         619→3. The leaf with the most recent `timestamp` is typically the active branch endpoint
         620→4. Trace backwards via `parentUuid` to reconstruct the active conversation
         621→
         622→### Leaves
         623→
         624→A "leaf" is a message whose `uuid` is not referenced as any other message's `parentUuid`. Sessions commonly have multiple leaves representing abandoned branches.
